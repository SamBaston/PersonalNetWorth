# PersonalNetWorth
Personal Net Worth and Debt tracker

## This branch

For this branch we used Claude AI's sonnet 4.6 model and used the following prompt to implement additional 'Asset Tracking' functionality on top of the 'Basic Start' branch that we agreed was best achieved by Z.ai:

```
Here are the files composing of a basic start to a web application. You are a senior full stack developer. I would like you to implement an 'Asset tracking' functionality following this criteria:

Bank Accounts: Support for multiple accounts (Current, Savings) with manual balance entry and currency support.
ISAs & LISAs: * Tracking for UK tax-advantaged accounts.
LISA logic: Include government bonus (25%) tracking as a separate pending or realized asset.
Stock Portfolios:
Input for individual ticker symbols or total portfolio value.
Integration for real-time (or delayed) price fetching.
```

When I unzipped, Claude's response, I was initially very impressed with it's additions; it provided functionality for all of the above critera in an ergonomic way, allowing the user to customise their assets and liabilities as they wish. There were, however, a few things which I wasn't happy with:

1.  While the user could add and delete stocks, the functionality which was intended to fetch real-time stock prices was broken, and I   was recieving the error:

    ```
    Failed to refresh AAPL: Failed to parse data for AAPL: Unexpected token T in JSON at position 0
    Failed to refresh NVDA: Failed to parse data for NVDA: Unexpected token T in JSON at position 0
    ```
    I gave Claude the following prompt in order to resolve this
    ```
    The code you have given me returns this error 

    Failed to refresh AAPL: Failed to parse data for AAPL: Unexpected token T in JSON at position 0
    Failed to refresh NVDA: Failed to parse data for NVDA: Unexpected token T in JSON at position 0 

    can you resolve this
    ```
    and it saw it fit to incorporate the 'Finnhub' api rather than 'YahooFinance'. After rewriting **fetchStockPrice** to accomodate for the new api, I ran the updated code and it worked as expected.

2.  The style of the user interface had been completely changed, interestingly to the same style as the application previously generated by Claude in the 'One-shot' attempt. While this isn't inherenly an issue, I do think that if you are uploading a W.I.P. project and instructing the model to make additions, it should take the initiatve to replicate the current style of the wesbite you give it rather than overwriting it. This highlights a clear bias in style of apps generated by the model and will be something that I make sure to discourage in my future prompts
